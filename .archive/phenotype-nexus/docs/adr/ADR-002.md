## ADR-002: Control Plane Architecture

**Status**: Accepted

**Date**: 2026-04-04

**Context**: The control plane manages service discovery, proxy configuration, security policies, and observability data. We must select an architecture that balances scalability, consistency, and operational simplicity. The control plane is the brain of the service mesh - it receives state from the cluster, computes policy, and distributes configuration to data plane proxies.

Key requirements:
- Support for 10,000+ services across multiple clusters
- Configuration propagation within 10 seconds
- High availability with no single point of failure
- Kubernetes-native CRD-based configuration
- GitOps-compatible workflow support

The control plane must handle:
- Service registry with health checking
- Certificate management and rotation
- Traffic routing policies
- Network security policies
- Observability data aggregation

## Options Considered

### Option 1: Centralized etcd-based Control Plane

**Description**: Traditional centralized architecture using etcd cluster for distributed configuration storage, with dedicated control plane components for policy computation and distribution.

**Pros**:
- Proven technology (used by Kubernetes, etcd, CoreDNS)
- Strong consistency guarantees (Raft consensus)
- Horizontal scalability via etcd federation
- Mature tooling and operational knowledge
- Well-understood failure modes

**Cons**:
- Single logical cluster creates blast radius
- etcd resource requirements grow with service count
- Configuration propagation requires multiple hops
- Complex upgrade procedures
- Monitoring and debugging complexity

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Max services | 50,000 | etcd performance tests |
| Config propagation | 5-15s | Kubernetes benchmarks |
| etcd memory (10K keys) | 2GB | Production metrics |
| Single point of failure | Yes | Architecture design |

### Option 2: Distributed Control Plane (Per-Cluster)

**Description**: Each cluster runs its own control plane with synchronization for multi-cluster features. Federation layer coordinates cross-cluster policies.

**Pros**:
- No single point of failure
- Lower latency for local operations
- Easier to reason about per-cluster state
- Independent upgrade cycles per cluster
- Better isolation between tenants

**Cons**:
- Cross-cluster operations require federation layer
- Inconsistent global state during partitions
- More complex deployment and operations
- Federation protocol complexity
- Debugging across clusters challenging

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Local propagation | 1-3s | Per-cluster measurements |
| Cross-cluster propagation | 10-30s | Federation benchmarks |
| Failure isolation | Complete | Architecture design |
| Operational complexity | High | Multiple control planes |

### Option 3: Kubernetes-native CRD + Operator

**Description**: Leverage Kubernetes extension mechanisms (CRDs, Operators, admission controllers) for all control plane functionality. No separate control plane processes.

**Pros**:
- Kubernetes-native integration
- Leverages existing Kubernetes tooling
- Declarative configuration via kubectl
- Built-in HA with Kubernetes
- Simplifies deployment and upgrades

**Cons**:
- Limited to Kubernetes environments
- Custom controller development complexity
- Resource usage for controllers
- Debugging CRD interactions challenging
- May not support hybrid workloads

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| CRD operation latency | 10-50ms | Kubernetes metrics |
| Controller reconciliation | 1-5s | Operator benchmarks |
| Memory overhead | 256MB per controller | Production metrics |
| Scalability limit | 10,000 CRs | CRD documentation |

### Option 4: Hybrid Architecture (Federated CRDs + Distributed Sync)

**Description**: Kubernetes-native CRDs for configuration with federation layer for cross-cluster synchronization. Local control plane components for real-time operations.

**Pros**:
- Best of both worlds: local speed + global consistency
- Kubernetes-native configuration model
- Federation for cross-cluster policies
- Gradual rollout of federation features
- Supports hybrid environments (K8s + VMs)

**Cons**:
- Most complex architecture to implement
- Multiple synchronization mechanisms
- Conflict resolution complexity
- Higher operational overhead
- Requires careful design for consistency

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Local operations | 1-3s | Distributed control plane |
| Cross-cluster sync | 5-15s | Federation benchmarks |
| Consistency model | Eventual | Federation design |
| Failure isolation | Partial | Hybrid architecture |

---

## Decision

**Chosen Option**: Option 4 - Hybrid Architecture (Federated CRDs + Distributed Sync)

**Rationale**:
1. **Scalability**: Per-cluster control planes handle local operations without single-point-of-failure concerns
2. **Performance**: Local operations complete in 1-3 seconds vs 10+ seconds for centralized approaches
3. **Kubernetes-native**: Leverages CRDs and operators for configuration management
4. **Multi-cluster**: Federation layer enables consistent policies across clusters when needed
5. **Resilience**: Cluster-level failures don't cascade to other clusters
6. **Hybrid support**: Can extend to non-Kubernetes workloads via federation adapters

**Evidence**:
- Google Anthos and VMware Tanzu use similar patterns for multi-cluster management
- Cilium's clustermesh architecture demonstrates federated control plane viability
- Kubernetes operator pattern proven for complex orchestration
- Industry trend toward distributed control planes for scale

**Alternatives Considered but Rejected**:
- **Centralized etcd**: Rejected due to single point of failure and scaling limitations
- **Pure distributed**: Rejected due to operational complexity without Kubernetes-native benefits
- **Kubernetes-only CRD**: Rejected due to limited hybrid/multi-cluster support

## Consequences

### Positive

- **Scalability**: Per-cluster control planes scale independently
- **Resilience**: Cluster failures don't cascade
- **Performance**: Local operations are fast (1-3s)
- **Kubernetes-native**: Uses familiar CRD patterns
- **Multi-cluster**: Federation enables global policy consistency
- **Hybrid support**: Can extend to non-K8s workloads

### Negative

- **Complexity**: Hybrid architecture is more complex to implement
- **Sync overhead**: Cross-cluster operations add latency
- **Debugging**: More components to trace for issues
- **Consistency tradeoffs**: Eventual consistency may cause brief inconsistencies
- **Operational burden**: Multiple synchronization mechanisms need monitoring

### Neutral

- **Migration path**: Existing clusters can join federation incrementally
- **Vendor flexibility**: Federation adapters can support multiple environments
- **Customization**: Per-cluster overrides for local requirements
