## ADR-004: Observability Architecture

**Status**: Accepted

**Date**: 2026-04-04

**Context**: Observability is critical for operating distributed systems effectively. phenotype-nexus must provide comprehensive observability covering metrics, logs, and traces with correlation across all data types. The observability stack must integrate with existing enterprise tools while providing unified views across services and clusters.

Key observability requirements:
- Metrics: Prometheus-compatible with SLI/SLO tracking
- Traces: OpenTelemetry-native distributed tracing
- Logs: Structured logging with correlation IDs
- Correlation: Cross-cutting concerns (trace ID in logs/metrics)
- Alerts: AlertManager-compatible alerting
- Dashboards: Grafana dashboards for all components

Observability architecture challenges:
- High cardinality from many service instances
- Volume of data at scale (10K+ services)
- Latency sensitivity of real-time metrics
- Storage costs for long retention
- Query performance for aggregations

## Options Considered

### Option 1: Traditional ELK Stack

**Description**: Elasticsearch, Logstash, Kibana for log aggregation with separate Prometheus + Grafana for metrics and Jaeger for tracing.

**Pros**:
- Mature, well-understood technology
- Strong log search capabilities
- Large community and documentation
- Good Kibana visualizations
- Flexible indexing strategy

**Cons**:
- Elasticsearch licensing changes (SSPL)
- Operational complexity (many components)
- High resource requirements
- Not optimized for metrics/traces
- Data correlation requires additional work
- Cost at scale

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Log ingestion rate | 100K logs/sec | ELK benchmarks |
| Search latency P99 | 500ms | Production metrics |
| Storage per million logs | 50GB | Documentation |
| Elasticsearch memory | 32GB+ recommended | Ops guide |

### Option 2: Grafana OSS Stack (LGTM)

**Description**: Loki for logs, Grafana for visualization, Prometheus for metrics, and Tempo for traces - all from Grafana Labs.

**Pros**:
- Open source with commercial support option
- Tight integration between components
- Loki is cost-effective for logs
- Prometheus is industry standard for metrics
- Tempo provides distributed traces
- Grafana is best-in-class visualization

**Cons**:
- Multiple components to operate
- Loki query performance at high volume
- Prometheus scaling challenges
- Correlation requires configuration
- No native alerting in Loki
- Talent availability may be limited

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Loki ingestion | 50K streams/sec | Loki benchmarks |
| Log storage cost | 5x cheaper than Elasticsearch | Grafana comparison |
| Prometheus query latency | 100ms P99 | Production metrics |
| Tempo queries | 200ms P99 | Grafana documentation |

### Option 3: Vendor-managed Observability (Datadog/New Relic)

**Description**: Full SaaS observability platform handling all data types with built-in correlation and analytics.

**Pros**:
- Zero operational overhead
- Built-in correlation and analytics
- Excellent user experience
- Automatic instrumentation
- No capacity planning needed
- Strong APM capabilities

**Cons**:
- Vendor lock-in
- High cost at scale
- Data residency concerns
- Limited customization
- Network latency for data access
- Compliance complexity

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Agent overhead | 1-3% CPU | Vendor documentation |
| Ingestion latency | <5s | Datadog SLA |
| APM overhead | 1-5% | Vendor benchmarks |
| Cost per host/month | $15-30 | Vendor pricing |

### Option 4: OpenTelemetry-native with Adaptive Storage

**Description**: OpenTelemetry for instrumentation and data collection with flexible backend storage (Prometheus, Loki, Tempo, or vendor backends) based on data type and retention requirements.

**Pros**:
- Vendor-neutral instrumentation
- Single SDK for all signals
- Portable across backends
- Future-proof architecture
- Strong industry momentum (CNCF)
- Consistent data model

**Cons**:
- OTel collector adds operational layer
- Backend diversity can be complex
- Correlation requires explicit configuration
- Best practices still evolving
- Some backend limitations
- Talent availability

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| OTel SDK overhead | <1% CPU | OTel benchmarks |
| Collector throughput | 10K spans/sec | OTel documentation |
| Context propagation | <1ms | Production metrics |
| Cardinality handling | Up to 1M series | Prometheus design |

---

## Decision

**Chosen Option**: Option 4 - OpenTelemetry-native with Adaptive Storage

**Rationale**:
1. **Vendor neutrality**: OTel provides portable instrumentation that isn't tied to specific backends
2. **Industry standard**: OpenTelemetry is rapidly becoming the de facto standard for observability
3. **Flexibility**: Adaptive storage allows optimization for different data types and cost requirements
4. **Future-proof**: OTel has strong momentum (CNCF graduated project) with all major vendors committing support
5. **Unified model**: Single SDK and data model for metrics, traces, and logs
6. **Enterprise integration**: Works with existing Prometheus/Grafana investments while enabling vendor backends when needed

**Evidence**:
- OpenTelemetry adoption at 65% in enterprises (CNCF Survey 2025)
- All major APM vendors (Datadog, New Relic, Dynatrace) supporting OTel
- 500+ contributors to OTel project
- Graduated CNCF project status achieved

**Alternatives Considered but Rejected**:
- **Traditional ELK**: Rejected due to Elasticsearch licensing concerns and operational complexity
- **Grafana OSS Stack**: Rejected due to multiple separate components and correlation challenges
- **Vendor-managed**: Rejected due to vendor lock-in and cost at scale

## Performance Benchmarks

```bash
# Observability stack benchmarking
# Requires: kubectl, prometheus, grafana, tempo, loki

# Deploy observability stack
helm repo update
helm install prometheus prometheus-community/prometheus
helm install grafana grafana/grafana
helm install loki grafana/loki
helm install tempo grafana/tempo

# Generate test load
for i in {1..100}; do
  kubectl run load-gen-$i --image=busybox -- \
    sh -c "while true; do wget -q -O- http://httpbin.default/get; done" &
done

# Metrics collection test
kubectl exec -it prometheus-0 -n default -- \
  promtool query instant \
  'rate(istio_requests_total[5m])'

# Distributed tracing test
kubectl port-forward svc/tempo 3100:3100 &
curl -s http://localhost:3100/api/echo -H "traceparent: 00-abcdef1234567890abcdef1234567890-01-01"

# Log aggregation test
kubectl logs -l app=httpbin --tail=1000 | \
  loki --stdin --addr=http://loki:3100

# Query performance test
time curl -s "http://loki:3100/loki/api/v1/query_range" \
  -d '{"query":"{app=\"httpbin\"}","start":1609459200,"end":1609459300}' \
  | jq '.data.result | length'
```

## Consequences

### Positive

- **Vendor neutrality**: Not locked into specific observability vendor
- **Industry standard**: OTel skills transfer across organizations
- **Cost flexibility**: Can mix open source and commercial backends
- **Unified data model**: Consistent instrumentation across all services
- **Future-proof**: OTel is the direction the industry is moving
- **Integration**: Works with existing investments (Prometheus, Grafana)

### Negative

- **Operational complexity**: Multiple components need management
- **Configuration overhead**: OTel collector requires careful tuning
- **Debugging**: Correlation issues can be complex to trace
- **Performance tuning**: Cardinality management is challenging
- **Learning curve**: Team needs OTel expertise
- **Early maturity**: Some OTel components still evolving

### Neutral

- **Migration path**: Can migrate from existing instrumentation incrementally
- **Backup strategies**: Different backends have different approaches
- **Cost model**: Can optimize based on data importance
