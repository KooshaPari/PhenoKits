KEY FINDINGS
============

PROBLEM:
  When launching DINO via launch-game command or test harness repeatedly,
  game fails with "Another instance is already running" fatal error.
  Blocks dev iteration, automated testing, CI/CD pipelines.

ROOT CAUSE:
  Win32 named mutexes (Global\Valve_SteamMutex_4199_*) created by Steam/DINO
  at game startup. Kernel doesn't immediately release mutex after process kill,
  preventing new instance from acquiring it.

SOLUTION:
  Hybrid 3-tier approach with graceful fallback:

  Tier 1: P/Invoke mutex release (primary, fast)
    • Enumerate and find DINO mutex
    • Call OpenMutex() + ReleaseMutex()
    • Direct kernel cleanup, most reliable if it works

  Tier 2: Extended sleep (fallback, proven)
    • Wait 10-15 seconds instead of 3
    • Allow natural kernel cleanup
    • Guaranteed to work, just slower

  Tier 3: Copy exe workaround (last resort)
    • Copy game exe to temp file
    • Launch copy (may bypass mutex checks)

EFFORT: 4-6 hours total
  Investigation:   1-2 hours (identify mutex, PoC P/Invoke)
  Implementation:  1 hour    (code 3 PowerShell functions)
  Testing:         1-2 hours (unit + integration + manual)
  Documentation:   30 mins   (update launch-game.md, docs)
  Release:         15 mins   (final commit)

RISK: LOW
  - Win32 APIs are stable and industry-standard
  - Multiple fallbacks ensure robustness
  - No impact to game logic or production
  - Pure dev/testing infrastructure improvement
