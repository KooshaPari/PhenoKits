================================================================================

EXECUTIVE SUMMARY
=================

The known-good commit (df3b55e) uses a SIMPLE, PROVEN approach:
  - 588 lines total, ~80 lines in Plugin.Awake()
  - Zero Harmony patches
  - Zero background threads
  - Zero resurrection logic
  - RESULT: Stable for weeks, no hangs on Windows

The current version (main) adds COMPLEX RESURRECTION INFRASTRUCTURE:
  - 1100+ lines total, ~300 lines in Plugin.Awake()
  - Four Harmony patches (PlayerLoop.SetPlayerLoop is the killer)
  - One background thread with Win32 polling
  - Five per-frame resurrection hooks
  - RESULT: Critical hang risk on Windows, especially during dotnet build and scene transitions

The hangs are caused by PlayerLoopSetPlayerLoopPatch:
  - DINO calls SetPlayerLoop during ECS bootstrap
  - Postfix fires, calls InjectPlayerLoopDrain()
  - InjectPlayerLoopDrain() rebuilds PlayerLoop via reflection
  - On Windows: GC lock contention during tight loops
  - In scene transitions: dozens of cascading calls
  - CUMULATIVE: Build hangs for 30-120 seconds

================================================================================
ROOT CAUSE OF HANGS
================================================================================

Trigger: dotnet build or scene transitions

Chain of events:
1. DINO ECS bootstrap calls PlayerLoop.SetPlayerLoop()
2. PlayerLoopSetPlayerLoopPatch.Postfix fires
3. Postfix calls Plugin.InjectPlayerLoopDrain()
4. InjectPlayerLoopDrain() rebuilds entire PlayerLoop via reflection
5. On Windows: reflection on PlayerLoop causes GC lock contention
6. Thread blocks waiting for GC to release locks
7. Build/transition thread times out
8. User experiences 30-120 second hang

Why Linux doesn't hang:
- Linux GC has different locking strategy
- Reflection lock contention less pronounced
- Same code path, different timing behavior

Why scene transitions are especially bad:
- DINO calls SetPlayerLoop multiple times
- Each call triggers postfix reflection
- Cumulative effect: dozens of calls in tight loop
- Hangs stack up

Why Windows is affected:
- .NET runtime on Windows uses different sync primitives
- GC lock holds player thread
- Reflection on PlayerLoop needs GC lock
- Deadlock-like behavior

================================================================================
THE DECISION: SIMPLE VS AMBITIOUS
================================================================================

OLD (df3b55e): Simple approach (accept graceful degradation)
  - Assume HideAndDontSave prevents destruction
  - If destruction happens (rare), accept it
  - NO Harmony patches, NO background threads
  - COST: F9/F10 don't work at menu after rare destruction
  - BENEFIT: No hangs, simple code, easy to debug

NEW (main): Ambitious approach (maximize F9/F10 availability)
  - Detect destruction, resurrect automatically
  - Use 4 Harmony patches, 1 background thread, 5 per-frame hooks
  - Ensure F9/F10 work even after destruction
  - COST: Hangs on Windows, complex code, platform-specific code
  - BENEFIT: F9/F10 work at menu in rare scenario

The new version violates vibecoding principle:
  Simple, proven > Ambitious, novel
