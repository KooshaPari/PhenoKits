# 2026-03-20 - Runtime Integration Regression Fix

## Summary

Investigated and fixed critical F9/F10 overlay regression where the runtime root GameObject was being destroyed at frame ~6s during scene transitions. Root cause traced to Harmony patch interference with DINOForge's two-boot cycle.

## Investigation

### Symptoms
- F9 (debug overlay) and F10 (mod menu) were working in-game but then UI disappeared after ~6 seconds
- Root GameObject marked with HideAndDontSave was still being destroyed
- Happened consistently during scene loads and within ~1 frame of scene transition completion

### Root Cause Analysis
The runtime used two Harmony patches to manage lifecycle state:
1. **LazyPatch** - intercepted GameObject creation with custom logic
2. **DeltaTimeResurrectionPatch** - tried to resurrect destroyed objects

These patches created a conflict:
- Harmony patch state leaked across scene boundaries
- DeltaTimeResurrectionPatch logic fought with native DontDestroyOnLoad behavior
- The framework's own persistence mechanism was being overridden by patch logic
- When scenes changed, patch conditions misaligned with framework state

### Two-Boot Cycle Context
DINO's Doorstop pre-loader initializes **twice** per playthrough:
- Boot 1: Initial load (game start)
- Boot 2: Scene transition (new game/continue)

The runtime must survive both boots without double-initialization. The Harmony patches attempted to manage this manually, but added complexity that broke under scene transitions.

## Solution

**Commit**: df3b55e (restore F9/F10 via RuntimeDriver.Update + immediate OnDestroy resurrection)

### Changes Made
1. **Removed all Harmony patches** from the codebase
2. **Restored pure persistence mechanism**:
   - Mark runtime root with `HideAndDontSave`
   - Add `DontDestroyOnLoad` to prevent garbage collection
   - RuntimeDriver.OnDestroy() resurrects root if accidentally destroyed
   - ModPlatform singleton ensures no double-initialization

### Why This Works
- **HideAndDontSave** prevents the scene serializer from destroying the root
- **DontDestroyOnLoad** is a native Unity flag, immune to scene reloads
- **DontDestroyOnLoad marker object** persists even if the main root is temporarily destroyed
- RuntimeDriver.OnDestroy() can detect and repair transient damage
- ModPlatform.Initialize checks for existing instance on Boot 2, skips re-initialization

## Lessons

1. **Patch state leakage** - Harmony patches that manage lifecycle state should be avoided; prefer native Unity mechanisms where possible
2. **Two-boot patterns** - When frameworks initialize twice, manually manage that in code, not through patches
3. **Trust native flags** - HideAndDontSave + DontDestroyOnLoad are proven; layering custom logic on top adds failure modes
4. **Observability** - Early problem detection with overlay visibility made root cause obvious once we traced the destruction timeline
