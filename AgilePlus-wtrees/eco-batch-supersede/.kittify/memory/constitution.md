## Technical Standards

### Languages and Frameworks

- **Rust**: Nightly (2024 edition). Primary language for core, agents, and integrations services.
  - Key crates: clap (CLI), axum + tokio (API), tonic + prost (gRPC), rusqlite (SQLite), git2 (libgit2), opentelemetry, serde, sha2, thiserror
- **Python**: 3.14+ (free-threaded mode enabled). MCP server and scripting.
  - Key packages: fastmcp>=3.0, grpcio, opentelemetry-sdk, pytest, hypothesis
- **TypeScript**: 7.0+ (native Go compiler). Use where it outperforms Python — web UI components, tooling scripts, or MCP extensions where TS ecosystem is stronger.
- **Protobuf**: buf-managed `.proto` files as the single source of truth for all inter-service contracts.
- **Open to**: Zig, Go for specific performance-critical components where Rust FFI overhead is unacceptable. Must justify in ADR.

- **Contract tests**: Pact contract tests for all gRPC boundaries (Rust↔Python, core↔agents, core↔integrations).
- **Integration**: Docker Compose test harness spinning up all 4 services.

## Tribal Knowledge

### Team Conventions

**Rust idioms (non-negotiable):**
- Use `Result<T, E>` for all fallible operations. Never `unwrap()` or `expect()` in production code.
- `#[must_use]` on all functions returning `Result` or important values.
- No `unsafe` without an `// SAFETY:` audit comment explaining why it's sound.
- `thiserror` for library errors, `anyhow` only in binary crates (CLI main).
- Newtype pattern for domain IDs (`FeatureId(i64)`, `WpId(i64)`) — no raw primitives in domain signatures.
- Prefer iterators over explicit loops where readability is maintained.
- Builder pattern for structs with >4 fields or optional configuration.

**Domain crate zero-dependency rule:**
- `agileplus-domain` crate has ZERO external dependencies except `serde`, `sha2`, and `chrono`.
- All I/O goes through port traits. Domain never touches network, filesystem, or database directly.
- This is enforced by Cargo.toml — domain crate's `[dependencies]` section is audited in CI.

**Agent-first design:**
- Every feature must be invocable programmatically (MCP tool, gRPC call, or SlashCommand).
- If an agent can't invoke it, it's not ready for release.
- Human CLI is a thin wrapper over the same programmatic interface.

### Lessons Learned

- **Scope creep prevention**: The multi-repo decomposition exists specifically to prevent scope creep. Every new capability must justify which repo it belongs to. If it doesn't fit any existing repo, it's either a new repo (with ADR justification) or out of scope.
- **Dependency minimalism**: Every dependency is a liability. Prefer std library. Audit all transitive deps with `cargo-audit` and `pip-audit`. Pin major versions. Review changelogs before upgrading.
- **Agent-first design**: Design for agent consumption first, human second. The agent workflow is the primary use case; the human CLI is a convenience wrapper.

### Historical Decisions

| Decision | Rationale | Alternative Rejected |
|----------|-----------|---------------------|
| Multi-repo (5 repos) over monorepo | Enforces service boundaries at repo level, prevents scope creep, enables independent versioning/deployment/CI | Monorepo — coupled releases, single CI bottleneck, scope creep |
| SQLite over PostgreSQL | Local-first simplicity, single-file database, rebuild from git if corrupted, no server dependency | PostgreSQL — requires server process, cloud dependency, overkill for single-dev/small-team |
| Python FastMCP over Rust MCP | MCP SDK is Python-first (FastMCP 3.0). Separate process over FFI/PyO3 for reliability and debuggability | PyO3 embedding — fragile, complicates deployment, harder to debug |

## Governance

### Amendment Process

- Any developer can propose amendments via pull request to this file.
- The `retrospective` command auto-suggests amendments based on feature learnings (data-driven).
- Coderabbit reviews constitution PRs for consistency and completeness.
- Amendments take effect immediately upon merge — no waiting period.

### Compliance Validation

- **Automated**: AgilePlus governance gates automatically validate constitution compliance at state transitions (specify → plan → implement → validate → ship). Violations block progression.
- **Coderabbit**: Constitution rules are encoded in Coderabbit configuration. PR reviews check for constitution violations (unwrap in prod, missing docs, etc.).
- **Human override**: Blocked transitions can be overridden with explicit justification recorded in the audit log as a governance exception.
- **Time-boxed exceptions**: Exceptions that bypass constitution rules have a sunset date. Auto-flagged for review after expiry during the next retrospective.

### Exception Handling

- Exceptions to the constitution are recorded as governance exceptions in the audit log.
- Each exception requires: justification, scope (which rule is being bypassed), and expected duration.
- Exceptions are auto-surfaced during `retrospective` for review.
- Recurring exceptions (same rule bypassed 3+ times) trigger a constitution amendment proposal — if everyone's breaking the rule, the rule should change.
- Exceptions without sunset dates are flagged as permanent overrides and require ADR documentation.
