# ADR-001: HexaKit Shelf Organization with Git Worktrees

__Status:__ Accepted  
__Date:__ 2026-04-04  
__Author:__ Platform Architect  

---

## Context

HexaKit contains approximately 30 independent git repositories under `CodeProjects/Phenotype/repos`. These repositories include:

- Infrastructure crates (phenotype-*)
- Application projects (AgilePlus, thegent, heliosCLI)
- Language kits (HexaPy, HexaGo, HexaType)
- Documentation and templates

The key challenge: How to organize ~30 independent repositories while maintaining:
1. Clear boundaries between projects
2. Efficient cross-project navigation
3. Consistent tooling and quality gates
4. AI agent context preservation

### Alternatives Considered

| Approach | Pros | Cons |
|----------|------|------|
| Monorepo (single repo) | Unified builds, atomic cross-project changes | Scale limits, IDE degradation, high switching cost |
| Polyrepo (30 separate clones) | Clear isolation, standard git workflow | Context switching, dependency drift, no unified view |
| Meta-repo (git-repo) | Coordinated operations | Adds complexity layer, limited tooling |
| __Shelf + Worktrees__ | Best of both worlds, minimal overhead | Requires discipline, worktree management |

## Decision

Adopt __Shelf Organization with Git Worktrees__ as the primary repository management strategy.

### Shelf Definition

A "shelf" is a directory containing related but independent git repositories, organized for discoverability and coordinated tooling.

```
HexaKit/
├── phenotype-infrakit/     # Foundation crates
├── agileplus/              # Spec-driven development engine
├── platforms/
│   └── thegent/            # Agent execution platform
├── heliosCLI/              # CLI harness
├── kits/
│   ├── HexaPy/
│   ├── HexaGo/
│   └── HexaType/
└── docs/                   # Documentation
```

### Worktree Strategy

Each project uses git worktrees for branch isolation:

```
agileplus/
├── .git/                   # Bare repository
├── main/                   # Main worktree (bare)
└── worktrees/
    ├── feat/auth-refactor/     # Feature worktree
    ├── fix/connection-leak/    # Fix worktree
    └── chore/update-deps/      # Maintenance worktree
```

### Worktree Operations

```bash
# Create feature worktree
git worktree add worktrees/feat/my-feature -b feat/my-feature

# Switch to worktree
cd worktrees/feat/my-feature

# Clean up
git worktree remove worktrees/feat/my-feature
```

## Consequences

### Positive

- __Parallel Development:__ Multiple features in progress simultaneously
- __Build Cache Preservation:__ Each worktree maintains its own target/ directory
- __Clean Boundaries:__ No cross-branch pollution
- __Standard Git:__ No custom tooling required
- __AI-Friendly:__ Each worktree is a complete context for agents

### Negative

- __Disk Usage:__ Multiple worktrees duplicate working directories
- __Complexity:__ Developers must understand worktree workflow
- __Tooling Gaps:__ Some tools assume single working directory
- __Navigation:__ Need conventions for worktree location

### Mitigations

| Concern | Mitigation |
|---------|------------|
| Disk usage | SSD storage is cheap; clean up old worktrees |
| Complexity | Document workflow; provide shell aliases |
| Tooling gaps | Wrapper scripts; IDE configuration |
| Navigation | Consistent naming; `worktrees/` convention |
