# ADR-002: AI-Native MCP Protocol Integration

__Status:__ Accepted  
__Date:__ 2026-04-04  
__Author:__ Platform Architect  

---

## Context

HexaKit requires integration with AI agents (Claude Code, Codex, Cursor, internal agents). The integration must:

1. Provide structured, type-safe interfaces
2. Enable tool discovery and invocation
3. Support resource exposure (specs, worklogs, codebase)
4. Work across language boundaries (Rust, Python, TypeScript)
5. Be locally operable (no cloud dependencies)

### Protocol Landscape (2024-2025)

| Protocol | Maturity | Type Safety | Local-First | Cross-Language |
|----------|----------|-------------|-------------|----------------|
| Custom JSON-RPC | High | Manual | Yes | Partial |
| LangChain Tools | Medium | Limited | Yes | Python-centric |
| OpenAI Functions | Medium | JSON Schema | Cloud | Yes |
| __MCP (Model Context Protocol)__ | Growing | JSON Schema | Yes | Yes |

MCP (Anthropic, 2024) is emerging as a standard for AI agent integration.

## Decision

Adopt __MCP (Model Context Protocol)__ as the primary AI integration protocol across all HexaKit projects.

### MCP Primitives

1. __Resources__ — URI-addressable content (specs, worklogs, code)
2. __Tools__ — Functions agents can invoke
3. __Prompts__ — Pre-defined prompt templates

### HexaKit MCP Servers

| Server | Language | Purpose |
|--------|----------|---------|
| `agileplus-mcp` | Rust | Spec management, planning |
| `thegent-mcp` | Python | Agent platform operations |
| `helios-mcp` | Rust | CLI integration |

### URI Conventions

```
kitty-spec://features/FEATURE-001         # Feature spec
kitty-spec://plans/PLAN-001               # Plan document
worklog://research/2026-04-04             # Research worklog
codebase://agileplus/src/main.rs          # Source file
evidence://features/FEATURE-001/tests   # Test evidence
```

### Transport Layer

| Transport | Use Case |
|-----------|----------|
| stdio | Local CLI tools |
| SSE | Remote/optional cloud |
| gRPC | Inter-service within HexaKit |

### Implementation Example

```rust
// Tool definition
#[derive(JsonSchema, Serialize, Deserialize)]
pub struct SpecifyArgs {
    pub title: String,
    pub description: String,
}

pub struct SpecifyTool {
    engine: Arc<SpecEngine>,
}

#[async_trait]
impl Tool for SpecifyTool {
    fn name(&self) -> String { "agileplus_specify".into() }
    
    fn schema(&self) -> Value {
        serde_json::to_value(SpecifyArgs::schema()).unwrap()
    }
    
    async fn call(&self, args: Value) -> Result<Value, ToolError> {
        let args: SpecifyArgs = serde_json::from_value(args)?;
        let spec = self.engine.specify(args.title, args.description).await?;
        Ok(serde_json::to_value(spec)?)
    }
}
```

## Consequences

### Positive

- __Standardization:__ Single protocol across all integrations
- __Type Safety:__ JSON Schema definitions
- __Discoverability:__ Agents can enumerate available tools/resources
- __Extensibility:__ New capabilities via new tools/resources
- __Ecosystem:__ Growing MCP adoption in agent ecosystem

### Negative

- __Immaturity:__ Protocol still evolving (2024-11 spec)
-__Tooling Gaps:__ Limited mature server implementations
-__Learning Curve:__ New concepts (resources, sampling)
-__Versioning:__ Need to track spec updates

### Mitigations

| Concern | Mitigation |
|---------|------------|
| Immaturity | Pin to specific spec version; abstraction layer |
| Tooling gaps | Build internal tooling; contribute upstream |
| Learning curve | Documentation; examples; training |
| Versioning | Automated spec compliance testing |
