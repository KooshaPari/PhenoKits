# ADR-003: Hierarchical Multi-Agent Organization

__Status:__ Accepted  
__Date:__ 2026-04-04  
__Author:__ Platform Architect  

---

## Context

HexaKit development involves multiple AI agents with different specializations. Without clear organization, agents may:

1. Overlap in responsibilities
2. Conflict on implementation decisions
3. Lose context between sessions
4. Create inconsistent output

### Agent Roles Identified

| Agent | Specialization | Primary Activities |
|-------|---------------|-------------------|
| Forge | Implementation | Coding, refactoring, scaffolding |
| Muse | Review | Quality, style, correctness analysis |
| Sage | Research | Investigation, analysis, documentation |
| Helios | Testing | Runtime verification, benchmarking |

### Coordination Challenges

- __Single-threaded work:__ Only one agent should own a task at a time
- __Handoff clarity:__ Context must transfer cleanly between agents
- __Conflict resolution:__ Disagreements need resolution protocol
- __User visibility:__ User must understand which agent is working

## Decision

Adopt __Hierarchical Multi-Agent Organization__ with primary-secondary relationships.

### Hierarchy Structure

```
User
 └── Forge (Primary)
      ├── Muse (Review)
      ├── Sage (Research)
      └── Helios (Testing)
```

### Primary Agent Responsibilities

1. __Task Ownership:__ Only one primary per task
2. __Coordination:__ Delegates to secondary agents as needed
3. __Final Output:__ Responsible for delivered result
4. __User Communication:__ Primary point of contact

### Secondary Agent Responsibilities

1. __Specialized Support:__ Provide expertise in domain
2. __No Direct Commits:__ Output goes through primary
3. __Advisory Role:__ Recommendations, not decisions
4. __Context Preservation:__ Document findings for handoff

### Delegation Protocol

```
Forge (primary) needs code review:

1. Forge: "Muse, please review this implementation"
2. Muse: Reviews, returns findings
3. Forge: Incorporates or disputes findings
4. If dispute: escalate to user
5. Final decision: documented in worklog
```

### Session Identification

Agents identify themselves at task start:

```
⏺ [HH:MM:SS] AgentName working on <project>:<task>

Example:
⏺ [14:32:10] Forge working on heliosCLI:auth-refactor
```

### State Transfer Format

```rust
pub struct AgentHandoff {
    pub from: AgentId,
    pub to: AgentId,
    pub task_id: Uuid,
    pub context_summary: String,
    pub decisions_made: Vec<Decision>,
    pub open_questions: Vec<String>,
    pub worklog_ref: String,
}
```

## Consequences

### Positive

- __Clear Responsibility:__ Each task has single owner
- __Specialization:__ Agents focus on strengths
- __Scalability:__ Can add new specialist agents
- __Auditability:__ Clear chain of custody
- __Conflict Reduction:__ Hierarchy resolves disputes

### Negative

- __Latency:__ Handoffs add coordination overhead
- __Context Loss:__ Information may not fully transfer
- __Bottleneck:__ Primary agent can become overloaded
- __Complexity:__ Users must understand agent roles

### Mitigations

| Concern | Mitigation |
|---------|------------|
| Latency | Async delegation; batch reviews |
| Context loss | Structured handoff format; worklogs |
| Bottleneck | Auto-delegation for routine tasks |
| Complexity | Clear documentation; role badges |
