# ADR-002: Dynamic Plugin Loading System

**Status**: Accepted

**Date**: 2026-04-04

**Deciders**: HelMo Architecture Team

---

## Context

CLI applications often need extensibility beyond compile-time configuration. The Phenotype ecosystem includes tools with varying plugin needs: some require runtime extension for custom commands, others need external integrations loaded on-demand. We evaluated plugin architectures for the agentkit framework that enable extensibility without recompilation.

Key requirements:
- Runtime loading of new commands without restart
- Sandboxed plugin execution
- Type-safe plugin interfaces
- Dependency management for plugins
- Clean uninstall/replacement

---

## Decision Drivers

| Driver | Priority | Notes |
|--------|----------|-------|
| Zero-Recompilation Extension | High | Add commands without rebuilding core binary |
| Type Safety | High | Plugin interfaces must be compile-time checked |
| Isolation | High | Plugin failures must not crash host application |
| Dependency Management | Medium | Plugins may have their own dependencies |
| Security | High | Plugins must be sandboxed from sensitive operations |

---

## Options Considered

### Option 1: Static Plugin Registration (Rejected)

**Description**: All plugins compiled into the main binary with registration macros.

**Pros**:
- Zero runtime overhead
- Full compiler optimization
- Simple debugging
- Type safety guaranteed

**Cons**:
- Requires recompilation for new plugins
- Binary size grows with plugins
- No third-party plugin ecosystem
- All plugins loaded at startup

**Assessment**: Suitable for fixed plugin sets but not for extensible frameworks.

### Option 2: Shared Library Plugins (dlopen)

**Description**: Plugins as `.so`/`.dylib` files loaded via `libloading` crate.

**Pros**:
- True runtime loading
- Can load/unload dynamically
- Language-agnostic (C ABI)
- Widely supported

**Cons**:
- No compile-time type checking of plugin interfaces
- Complex error handling (symbol resolution, ABI mismatches)
- Security concerns (arbitrary code execution)
- Dependency conflicts between plugins
- Debugging is difficult

**Performance Data**:

| Metric | Static | dlopen | wasmer | Notes |
|--------|--------|--------|--------|-------|
| Load time | 0ms | 15-50ms | 5-20ms | WASM faster |
| Memory isolation | None | None | Full | WASM wins |
| Type safety | Full | None | Full | Static/WASM |
| Uninstall complexity | Rebuild | File delete | File delete | Dynamic easier |

### Option 3: WebAssembly Plugin Runtime (Accepted)

**Description**: Plugins compiled to WASM, loaded via `wasmtime` or `wasmer`.

**Pros**:
- Strong sandboxing (memory, syscalls)
- Compile-time interface validation via WIT
- Language-agnostic (Rust, Go, C, JS, etc.)
- Deterministic execution
- Easy unloading (no DLL hell)
- Rich ecosystem (WASI, component model)

**Cons**:
- WASM MVP limitations (no threads, limited SIMD)
- Performance overhead (10-30% for some workloads)
- Larger binary if bundling runtime
- WASM tooling still maturing

**Performance Data**:

| Metric | Value | Notes |
|--------|-------|-------|
| Plugin load time | 5-20ms | Depends on plugin size |
| WASM->Host call | 1-5 microseconds | Per call overhead |
| Memory overhead | 2-5MB per plugin | WASM runtime |
| Cold start | 50-200ms | JIT compilation |

---

## Decision

**Chosen Option**: WebAssembly (wasmtime) for plugin isolation

**Rationale**:
1. **Security**: WASM sandbox prevents plugin code from accessing unauthorized resources.
2. **Type Safety**: WIT (WebAssembly Interface Types) provides compile-time interface validation.
3. **Language Agnosticism**: Plugin authors can use Rust, Go, Python, etc.
4. **Clean Unloading**: WASM modules can be dropped without DLL hell.
5. **Ecosystem**: WASI provides standardized system interface for plugins.

**Evidence**:
- wasmtime: https://github.com/bytecodealliance/wasmtime
- WIT spec: https://github.com/WebAssembly/component-model
- Fermyon plugin architecture: https://www.fermyon.com/spin

## Consequences

### Positive

- Third-party plugins without binary modification
- Strong isolation prevents plugin crashes affecting host
- Language-agnostic plugin development
- Easy plugin discovery and versioning
- Parallel development of core and plugins

### Negative

- WASM runtime adds binary size (~2-5MB)
- Some performance overhead for plugin calls
- WASM limitations (no direct filesystem access without WASI)
- Tooling complexity (WIT IDL, bindgen)

### Neutral

- Plugin authors need WASM knowledge
- Debugging requires WASM-specific tools
- Distribution is different (WASM modules vs DLLs)
