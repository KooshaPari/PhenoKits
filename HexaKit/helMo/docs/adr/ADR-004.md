# ADR-004: OpenTelemetry Observability Integration

**Status**: Accepted

**Date**: 2026-04-04

**Deciders**: HelMo Architecture Team

---

## Context

CLI applications in production environments require observability beyond simple logging. As agentkit powers more Phenotype tools, we need structured tracing, metrics, and logging that can integrate with enterprise observability platforms (Datadog, Honeycomb, Jaeger). We evaluated observability approaches for the framework.

Key requirements:
- Structured logging with correlation IDs
- Distributed tracing for distributed operations
- Metrics collection (counters, histograms, gauges)
- OpenTelemetry protocol support
- Zero-cost abstraction for disabled telemetry

---

## Decision Drivers

| Driver | Priority | Notes |
|--------|----------|-------|
| Vendor Neutrality | High | Teams use different observability platforms |
| Performance | High | Telemetry must not slow CLI operations |
| Correlation | High | Link spans across service boundaries |
| Standards | High | OpenTelemetry is the industry standard |
| Ergonomics | Medium | Easy to add instrumentation |

---

## Options Considered

### Option 1: Custom Logging + Metrics (Rejected)

**Description**: Build homegrown observability primitives.

**Pros**:
- No external dependencies
- Full control over implementation
- Simple initially

**Cons**:
- Reinventing the wheel
- No vendor integration
- No industry support
- Custom format = no tooling support
- Time investment in maintenance

**Assessment**: Technical debt for non-differentiated functionality.

### Option 2: Structured Logging Only (Rejected)

**Description**: Use structured logging (tracing + log) without distributed tracing.

**Pros**:
- Simpler than full OTEL
- Good for single-process apps
- Good ecosystem (tracing crate)

**Cons**:
- No distributed tracing
- No metrics standardization
- Limited correlation across services
- Not suitable for microservice architectures

**Assessment**: Insufficient for Phenotype's distributed tool ecosystem.

### Option 3: OpenTelemetry SDK (Accepted)

**Description**: Full OpenTelemetry integration with tracing, metrics, and logging.

**Pros**:
- Vendor-neutral standard
- Distributed tracing out of the box
- Rich ecosystem (OTLP exporters)
- Structured logs with span correlation
- Metrics with PromQL compatibility
- Active maintenance (CNCF)

**Cons**:
- Larger dependency footprint
- Configuration complexity
- Resource detection needs setup
- Sampling strategy considerations

**Performance Data**:

| Metric | No Telemetry | OTel Sampling | OTel Full | Notes |
|--------|--------------|---------------|-----------|-------|
| Latency overhead | 0ns | 50-200ns | 500-2000ns | Sampling reduces cost |
| Memory per span | 0KB | 1-5KB | 5-20KB | Depends on attributes |
| Export batch | N/A | 1-5ms | 1-5ms | Background export |
| CPU overhead | 0% | 0.1-0.5% | 1-3% | With 100% sampling |

---

## Decision

**Chosen Option**: OpenTelemetry SDK with OTLP exporter

**Rationale**:
1. **Standards**: OpenTelemetry is the CNCF standard for observability.
2. **Vendor Neutrality**: Export to any backend (Jaeger, Datadog, Honeycomb, etc.)
3. **Distributed Tracing**: Span correlation across service boundaries.
4. **Metrics**: Counters, histograms, gauges with PromQL compatibility.
5. **Ecosystem**: Growing Rust support via `opentelemetry-rust`.

**Evidence**:
- OpenTelemetry: https://opentelemetry.io/
- opentelemetry-rust: https://github.com/open-telemetry/opentelemetry-rust
- OTel Rust SDK: https://docs.rs/opentelemetry/latest/opentelemetry/

## Consequences

### Positive

- Industry-standard observability
- Vendor flexibility
- Distributed tracing correlation
- Rich metric collection
- Professional debugging capabilities

### Negative

- Additional dependencies (opentelemetry, tonic, prost)
- Configuration complexity
- Sampling strategy needed for cost control
- Runtime overhead even with sampling

### Neutral

- Requires observability platform for full benefit
- Training needed for team on OTel concepts
- Trace ID propagation requires explicit handling
