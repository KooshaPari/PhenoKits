# ADR-001: HTTP Client Architecture for Agent API

## Status: Accepted

**Date:** 2026-04-05  
**Author:** Phenotype Engineering Team  
**Project:** agentapi

---

## Context

The agentapi library requires an HTTP client to communicate with remote agent servers. We need to select an appropriate architecture that balances simplicity, performance, and maintainability.

### Requirements

1. **Context Support**: Must propagate cancellation and timeouts through request lifecycle
2. **Minimal Dependencies**: Reduce external dependency footprint
3. **Response Streaming**: Support for command output streaming
4. **API Key Authentication**: Simple header-based auth
5. **Error Handling**: Structured error responses

### Constraints

- Go 1.21+ (stdlib `slog` usage)
- No external HTTP client dependencies if possible
- Must work with standard `net/http`

---

## Decision

We will use a **thin wrapper around `net/http`** with the following architecture:

```
┌────────────────────────────────────────────────────────────────┐
│                      agentapi.Client                            │
│                                                                 │
│  ┌─────────────────┐        ┌─────────────────┐              │
│  │  Configuration  │───────▶│  net/http.Client │              │
│  │  - BaseURL      │        │                  │              │
│  │  - APIKey       │        │  - Timeout       │              │
│  │  - Headers      │        │  - Transport     │              │
│  └─────────────────┘        └─────────────────┘              │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Request Building                                         │   │
│  │  - Method, Path                                        │   │
│  │  - Headers (X-API-Key)                                   │   │
│  │  - JSON Body encoding                                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │  Response Handling                                        │   │
│  │  - Status code check                                     │   │
│  │  - JSON decoding                                         │   │
│  │  - Error mapping                                         │   │
│  └─────────────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────────────┘
```

### Rationale

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| **net/http wrapper** | Zero deps, stdlib, simple | Manual features | **SELECTED** |
| resty | Rich features, popular | Heavy dependency, overkill | Rejected |
| sling | Clean API | No streaming, extra dep | Rejected |
| go-resty | Production proven | Too many features | Rejected |

### Implementation

```go
type Client struct {
    BaseURL string
    APIKey  string
    client  *http.Client
}

func (c *Client) doRequest(ctx context.Context, method, path string, body interface{}) (*http.Response, error) {
    // Build URL
    url := c.BaseURL + path
    
    // Encode body
    var bodyReader io.Reader
    if body != nil {
        jsonBody, _ := json.Marshal(body)
        bodyReader = bytes.NewReader(jsonBody)
    }
    
    // Create request with context
    req, err := http.NewRequestWithContext(ctx, method, url, bodyReader)
    if err != nil {
        return nil, err
    }
    
    // Set headers
    req.Header.Set("Authorization", "Bearer "+c.APIKey)
    req.Header.Set("Content-Type", "application/json")
    req.Header.Set("Accept", "application/json")
    
    // Execute
    return c.client.Do(req)
}
```

## Consequences

### Positive

- **Zero external dependencies**: Only uses Go standard library
- **Full context support**: Native `context.Context` integration
- **Predictable behavior**: No hidden middleware or hooks
- **Easy to debug**: Straightforward request/response flow
- **Small binary size**: No bloated dependencies

### Negative

- **Manual retry logic**: Must implement ourselves if needed
- **No built-in metrics**: Need to add instrumentation
- **Limited middleware**: Must build interceptors manually
- **No automatic unmarshaling**: Manual JSON handling

### Mitigations

1. Add retry logic as separate wrapper function
2. Use OpenTelemetry for tracing/metrics
3. Implement simple middleware chain if needed
4. Helper functions for common response patterns
