# ADR-002: API Key Authentication Strategy

## Status: Accepted

**Date:** 2026-04-05  
**Author:** Phenotype Engineering Team  
**Project:** agentapi

---

## Context

The agentapi library needs to authenticate requests to the agent server. We need to select an authentication mechanism that balances security, simplicity, and compatibility.

### Threat Model

1. **Token theft**: API keys must be protected in transit and at rest
2. **Replay attacks**: Requests should be time-bound where possible
3. **Scope limitation**: Different keys for different agents

### Constraints

- Must work over HTTPS
- No complex OAuth flow (server-to-server only)
- Support for multiple authentication schemes (for future)

---

## Decision

We will use **HTTP Bearer Token (API Key)** authentication via the `Authorization` header.

```
┌────────────────────────────────────────────────────────────────┐
│              API Key Authentication Flow                        │
│                                                                 │
│  Client                      Server                             │
│    │                          │                                │
│    │──── GET /agent/123 ─────▶│                                │
│    │    Authorization:        │                                │
│    │    Bearer <api-key>      │                                │
│    │                          │──────▶ Validate API key          │
│    │                          │        against database         │
│    │                          │◀────── Return agent info       │
│    │◀──── 200 OK ─────────────│                                │
│    │     {id: 123, ...}       │                                │
│    │                          │                                │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ Key Characteristics:                                      │   │
│  │ • Simple header-based auth                                │   │
│  │ • Compatible with standard HTTP clients                   │   │
│  │ • Easy to rotate and revoke                               │   │
│  │ • Works with reverse proxy auth                           │   │
│  └─────────────────────────────────────────────────────────┘   │
└────────────────────────────────────────────────────────────────┘
```

### Rationale

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| **Bearer Token** | Simple, standard, widely supported | No built-in expiration | **SELECTED** |
| mTLS | Most secure | Complex setup, cert management | Future consideration |
| JWT | Self-contained, expirable | Larger tokens, overkill | Rejected |
| HMAC | Request signing | Complex client implementation | Rejected |
| Basic Auth | Simple | Credentials in every request | Rejected |

## Consequences

### Positive

- **Standard approach**: Compatible with most HTTP clients
- **Simple implementation**: Single header to set
- **Easy rotation**: Generate new key, update config
- **Proxy-friendly**: Works with API gateways
- **Debuggable**: Clear in request logs

### Negative

- **Token in headers**: Visible in access logs (mitigate with HTTPS)
- **No built-in expiration**: Manual key rotation required
- **No scope limitation**: Single key grants all permissions
- **Replay risk**: Token valid until revoked

### Mitigations

1. **HTTPS Required**: Never transmit over plain HTTP
2. **Key Rotation**: Document rotation procedure
3. **Short-lived keys**: Consider 90-day rotation policy
4. **Audit Logging**: Log all API key usage
5. **Rate Limiting**: Per-key request limits

## Security Considerations

```
┌────────────────────────────────────────────────────────────────┐
│              Security Checklist                               │
│                                                                 │
│  □ API keys generated with cryptographically secure RNG      │
│  □ Minimum key length: 32 bytes                               │
│  □ Keys stored hashed (bcrypt/argon2) on server              │
│  □ HTTPS enforced (HSTS header)                               │
│  □ Rate limiting per API key                                  │
│  □ Audit logging for all authenticated requests               │
│  □ Key rotation procedure documented                          │
│  □ Revocation endpoint available                              │
│  □ No API keys in URL or response bodies                     │
│  □ Keys not logged (redacted in logs)                         │
└────────────────────────────────────────────────────────────────┘
```
