# ADR-003: Error Handling Pattern

## Status: Accepted

**Date:** 2026-04-05  
**Author:** Phenotype Engineering Team  
**Project:** agentapi

---

## Context

The agentapi library needs a consistent error handling strategy for both client-side errors (network, serialization) and server-side errors (authentication, not found, server errors).

### Requirements

1. Distinguish between retryable and non-retryable errors
2. Preserve error context for debugging
3. Compatible with Go 1.21+ error handling patterns
4. JSON error responses from server

### Constraints

- Must work with `errors.Is` and `errors.As`
- Support error wrapping
- HTTP status code preservation

---

## Decision

We will use **typed errors with wrapping** following Go 1.13+ patterns.

```
┌────────────────────────────────────────────────────────────────┐
│              Error Type Hierarchy                               │
│                                                                 │
│  error (builtin)                                                │
│    │                                                            │
│    ├── ClientError                                              │
│    │     ├── ErrConnectionFailed  (network errors)             │
│    │     ├── ErrTimeout           (context deadline)           │
│    │     └── ErrSerialization     (JSON encode/decode)           │
│    │                                                            │
│    └── ServerError                                              │
│          ├── ErrUnauthorized     (401)                          │
│          ├── ErrForbidden        (403)                          │
│          ├── ErrNotFound         (404)                          │
│          └── ErrServerError      (5xx)                          │
│                                                                 │
└────────────────────────────────────────────────────────────────┘
```

### Rationale

| Pattern | Pros | Cons | Verdict |
|---------|------|------|---------|
| **Typed errors** | Type-safe, `errors.Is` support | More types to maintain | **SELECTED** |
| Sentinel errors | Simple | Limited context | Partial |
| Error strings | Simple | Brittle parsing | Rejected |
| Error codes | Parseable | Less Go-idiomatic | Rejected |
| Panic | Simple | Hard to recover | Rejected |

### Implementation

```go
// Sentinel errors for common cases
var (
    ErrConnectionFailed = errors.New("connection to agent failed")
    ErrTimeout          = errors.New("request timeout")
    ErrUnauthorized     = errors.New("authentication failed")
)

// ServerError captures HTTP error responses
type ServerError struct {
    StatusCode int
    Message    string
    RequestID  string
}

func (e *ServerError) Error() string {
    return fmt.Sprintf("server error %d: %s (request: %s)", 
        e.StatusCode, e.Message, e.RequestID)
}

// HTTPError creates appropriate error from response
func HTTPError(resp *http.Response) error {
    body, _ := io.ReadAll(resp.Body)
    
    switch resp.StatusCode {
    case http.StatusUnauthorized:
        return ErrUnauthorized
    case http.StatusNotFound:
        return fmt.Errorf("%w: %s", ErrNotFound, string(body))
    default:
        return &ServerError{
            StatusCode: resp.StatusCode,
            Message:    string(body),
        }
    }
}
```

---

## Consequences

### Positive

- **Type-safe error checking**: Use `errors.Is()` for specific errors
- **Error wrapping**: Preserve full error chain with `fmt.Errorf("%w", err)`
- **HTTP status preserved**: Can extract from `ServerError`
- **Retry logic**: Can check `errors.Is(err, ErrTimeout)` for retry
- **Go idiomatic**: Follows standard Go error patterns

### Negative

- **Type proliferation**: Multiple error types to maintain
- **Wrapping discipline**: Must consistently wrap errors
- **Documentation**: Need to document which errors each function returns

### Mitigations

1. Document error types in function comments
2. Provide helper functions: `IsRetryable(err error) bool`
3. Error code constants for programmatic handling

## Error Response Format

```go
// Server error response JSON
{
    "error": {
        "code": "AGENT_NOT_FOUND",
        "message": "Agent with ID '123' not found",
        "request_id": "req-abc-123",
        "timestamp": "2024-01-15T10:30:00Z",
        "details": {
            "agent_id": "123"
        }
    }
}

// Client parsing
func parseErrorResponse(body []byte) error {
    var resp struct {
        Error struct {
            Code    string                 `json:"code"`
            Message string                 `json:"message"`
            Details map[string]interface{} `json:"details"`
        } `json:"error"`
    }
    
    if err := json.Unmarshal(body, &resp); err != nil {
        return fmt.Errorf("failed to parse error: %w", err)
    }
    
    return &ServerError{
        Message: resp.Error.Message,
    }
}
```
