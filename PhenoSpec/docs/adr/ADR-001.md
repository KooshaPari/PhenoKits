# ADR-001: PhenoSpec - Specification Format and Schema

## Status

**Accepted**

## Context

The Phenotype ecosystem requires a standardized specification format for defining features, APIs, and requirements. PhenoSpec provides a machine-readable, version-controlled approach to specifications that enables automation, validation, and traceability.

Key challenges:

1. **Consistency**: Different teams use different formats
2. **Traceability**: Linking specs to code and tests
3. **Validation**: Ensuring specs meet quality standards
4. **Evolution**: Managing spec changes over time
5. **Collaboration**: Review and approval workflows
6. **Integration**: Connecting specs to other tools

## Decision

We will implement a structured specification format:

### 1. Specification Format

**Rationale**: Structured format enables tooling.

**Decision Details**:
- YAML-based for human readability
- JSON Schema for validation
- Markdown for narrative sections
- Semantic versioning for specs
- Unique identifier per specification
- Hierarchical organization (epic → feature → story)

### 2. Schema Definition

**Rationale**: Schema ensures consistency.

**Decision Details**:
- `id`: Unique identifier
- `title`: Human-readable name
- `status`: Draft, Review, Approved, Implemented, Deprecated
- `description`: Detailed explanation
- `acceptance_criteria`: List of verifiable criteria
- `dependencies`: Related specs
- `assignee`: Owner
- `priority`: Critical, High, Medium, Low
- `estimates`: Time/effort estimates

### 3. Validation Pipeline

**Rationale**: Automated validation ensures quality.

**Decision Details**:
- Schema validation on commit
- Link validation for dependencies
- Circular dependency detection
- Acceptance criteria completeness check
- Identifier uniqueness validation
- Status transition validation

### 4. Version Control Integration

**Rationale**: Specs should be versioned like code.

**Decision Details**:
- Specs stored in Git
- Branch-based workflow
- Pull request for spec review
- Approval required before merge
- Change history tracked in Git
- Diff visualization for spec changes

### 5. Tooling and Automation

**Rationale**: Automation reduces manual work.

**Decision Details**:
- `pheno spec init` for creating specs
- `pheno spec validate` for local validation
- `pheno spec link` for traceability linking
- CI/CD integration for automated checks
- Dashboard for spec status overview
- Notifications for status changes

## Consequences

### Positive

1. **Consistency**: Standard format across all specs
2. **Quality**: Validation catches errors early
3. **Traceability**: Clear lineage from spec to code
4. **Collaboration**: Git-based review workflow
5. **Automation**: Tools reduce manual overhead
6. **Visibility**: Dashboard shows spec status

### Negative

1. **Learning Curve**: New format to learn
2. **Overhead**: More structure than free-form docs
3. **Tooling**: Custom tools to build and maintain
4. **Migration**: Existing specs need conversion
5. **Rigidity**: Strict format may not fit all cases

### Neutral

1. **Storage**: YAML in Git is lightweight
2. **Rendering**: Multiple output formats possible
3. **Extensibility**: Schema can evolve

## Alternatives

### Alternative 1: Free-Form Markdown

**Considered**: Markdown specs without structure.

**Rejected Because**:
- No validation possible
- Inconsistent formatting
- Hard to automate
- No standard fields

### Alternative 2: JIRA/Linear Only

**Considered**: Use existing issue trackers.

**Rejected Because**:
- Vendor lock-in
- No version control integration
- Hard to link to code
- Limited customization

### Alternative 3: Proprietary Format

**Considered**: Binary or custom format.

**Rejected Because**:
- Vendor lock-in
- Hard to edit
- No ecosystem tooling

### Alternative 4: XML

**Considered**: XML-based specification format.

**Rejected Because**:
- Verbose
- Hard to read
- Less popular than YAML
