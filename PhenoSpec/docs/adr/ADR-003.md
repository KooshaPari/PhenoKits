# ADR-003: PhenoSpec Integration with Agile Tools

## Status

**Accepted**

## Context

Specifications must integrate with existing Agile workflows and tools. Bi-directional sync with issue trackers enables teams to work in their preferred tools while maintaining spec traceability.

Key challenges:

1. **Sync**: Keeping spec and issue tracker in sync
2. **Conflict**: Handling simultaneous edits
3. **Workflow**: Mapping spec status to issue status
4. **Fields**: Mapping custom fields
5. **Performance**: Efficient sync without overwhelming APIs
6. **Reliability**: Handling API outages

## Decision

We will implement bi-directional sync with popular Agile tools:

### 1. Supported Integrations

**Rationale**: Support teams where they work.

**Decision Details**:
- JIRA Cloud and Server
- Linear
- GitHub Issues
- GitLab Issues
- Azure DevOps
- Shortcut
- Integration via REST APIs and webhooks

### 2. Sync Strategy

**Rationale**: Conflict-free synchronization.

**Decision Details**:
- Spec is source of truth for requirements
- Issue tracker for sprint planning and assignment
- One-way sync: Spec → Issue on creation
- Bi-directional sync for status and comments
- Conflict resolution rules defined
- Manual sync trigger available

### 3. Field Mapping

**Rationale**: Meaningful data exchange.

**Decision Details**:
- Title ↔ Summary
- Description ↔ Description
- Status ↔ Status (with mapping rules)
- Priority ↔ Priority
- Assignee ↔ Assignee
- Labels ↔ Labels/Components
- Custom field mapping configurable

### 4. Workflow Integration

**Rationale**: Respect existing workflows.

**Decision Details**:
- Configurable status mappings
- Transition rules honored
- Workflow validation before updates
- Epic/story hierarchy preservation
- Sprint assignment sync optional
- Board column mapping

### 5. Reliability

**Rationale**: Sync must be reliable.

**Decision Details**:
- Queue-based sync processing
- Retry with exponential backoff
- Dead letter queue for failures
- Manual retry capability
- Sync status dashboard
- Health checks for integrations

## Consequences

### Positive

1. **Flexibility**: Teams use preferred tools
2. **Visibility**: Specs visible in issue trackers
3. **Efficiency**: No double data entry
4. **Traceability**: Links maintained automatically
5. **Adoption**: Lower barrier to spec adoption
6. **Workflow**: Existing workflows respected

### Negative

1. **Complexity**: Sync logic complex
2. **API Limits**: Rate limiting concerns
3. **Latency**: Sync not instantaneous
4. **Conflict**: Occasional merge conflicts
5. **Maintenance**: API changes require updates

### Neutral

1. **Vendor**: Dependent on API stability
2. **Scale**: Sync volume considerations
3. **Customization**: Per-team configuration

## Alternatives

### Alternative 1: Issue Tracker Only

**Considered**: Use issue tracker as spec source.

**Rejected Because**:
- No version control
- Limited structure
- Vendor lock-in
- Poor code linking

### Alternative 2: Spec Only

**Considered**: No issue tracker integration.

**Rejected Because**:
- Forces tool change on teams
- Poor sprint planning support
- Low adoption

### Alternative 3: One-Way Sync Only

**Considered**: Only sync spec to issue tracker.

**Rejected Because**:
- Updates in tracker lost
- Comments not synced
- Poor workflow integration

### Alternative 4: Import/Export

**Considered**: Manual import/export only.

**Rejected Because**:
- Manual overhead
- Out of sync quickly
- Error-prone
