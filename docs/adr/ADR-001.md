# ADR-001: Documentation Site Architecture - VitePress-Based Static Generation

## Status

**Accepted**

## Context

The Phenotype ecosystem requires a comprehensive documentation platform that can serve multiple projects with varying needs. The existing documentation was scattered across repositories, formats, and platforms, leading to fragmentation and maintenance overhead.

Key challenges identified:

1. **Fragmentation**: Documentation existed in Markdown files, wiki pages, Confluence, and Notion, making it difficult to maintain consistency.

2. **Versioning**: Multiple versions of APIs and libraries needed corresponding documentation versions, but no unified versioning strategy existed.

3. **Searchability**: Users struggled to find relevant information across the documentation landscape.

4. **Contribution Barrier**: The documentation contribution process was complex, discouraging developers from keeping docs updated.

5. **Multi-Format Output**: Documentation needed to support web viewing, PDF generation, and IDE integration.

6. **Integration Requirements**: The documentation system needed to integrate with CI/CD pipelines for automated deployments and validation.

## Decision

We have decided to adopt VitePress as the foundational documentation site generator with the following architectural decisions:

### 1. Static Site Generation with VitePress

**Rationale**: VitePress provides fast build times, excellent Vue-based theming capabilities, built-in search, and markdown-native authoring.

**Decision Details**:
- All documentation will be authored in Markdown
- VitePress will generate static HTML for deployment
- Custom themes will be developed for Phenotype brand consistency
- Built-in Vue components will enhance interactivity where needed

### 2. Monorepo Structure with Per-Project Documentation

**Rationale**: Keeping documentation close to code improves maintenance and discoverability.

**Decision Details**:
- Each project maintains its own `docs/` directory
- A central documentation engine aggregates content from project docs
- Cross-project links are validated during CI
- Shared components and styles are managed centrally

### 3. Automated Documentation Pipeline

**Rationale**: Documentation must stay in sync with code changes without manual overhead.

**Decision Details**:
- Pre-commit hooks validate markdown syntax
- CI pipeline runs link checking and spell checking
- Automated deployments to staging on PR
- Production deployment on merge to main
- Versioned documentation for released artifacts

### 4. Multi-Format Output Strategy

**Rationale**: Different use cases require different documentation formats.

**Decision Details**:
- Web: Primary VitePress-generated site
- PDF: Automated generation using Playwright + Paged.js
- IDE: LSP-compatible documentation for hover tooltips
- CLI: Man page generation from Markdown

### 5. Search Architecture

**Rationale**: Effective search is critical for documentation usability.

**Decision Details**:
- Built-in VitePress local search for smaller sites
- Algolia DocSearch integration for main documentation portal
- Structured data (JSON-LD) for SEO optimization
- Full-text search across all content types

## Consequences

### Positive

1. **Unified Experience**: Single, consistent documentation platform across all projects
2. **Developer Efficiency**: Markdown-native authoring reduces friction
3. **Performance**: Static generation provides excellent page load speeds
4. **SEO**: Structured markup and fast loading improve search rankings
5. **Maintainability**: Centralized theming and components reduce duplication
6. **CI Integration**: Automated quality checks prevent broken links and outdated content

### Negative

1. **Learning Curve**: Team must learn VitePress-specific features and Vue component development
2. **Build Time**: Large documentation sets may require significant build time
3. **Dynamic Content**: Static generation limits real-time content capabilities
4. **Migration Effort**: Existing documentation requires conversion and restructuring
5. **Vendor Lock-in**: Heavy reliance on VitePress/Vue ecosystem

### Neutral

1. **Vue Dependency**: Requires familiarity with Vue for custom components
2. **Node.js Requirement**: Build process depends on Node.js toolchain
3. **Markdown Extensions**: Must balance standard Markdown with VitePress-specific features

## Alternatives

### Alternative 1: Docusaurus

**Considered**: Docusaurus is a popular React-based documentation generator.

**Rejected Because**:
- React dependency conflicts with Phenotype's Vue preference
- Heavier bundle size compared to VitePress
- More complex configuration for our use case

### Alternative 2: MkDocs with Material Theme

**Considered**: Python-based static site generator with excellent Material Design theme.

**Rejected Because**:
- Python toolchain adds complexity to Node.js-focused projects
- Less flexibility for custom interactive components
- Smaller ecosystem for Vue-like reactive features

### Alternative 3: GitBook

**Considered**: Hosted documentation platform with WYSIWYG editing.

**Rejected Because**:
- Vendor lock-in and subscription costs
- Limited customization options
- Less control over CI/CD integration
- Difficult to version alongside code

### Alternative 4: Custom Solution

**Considered**: Building a custom documentation generator from scratch.

**Rejected Because**:
- Significant development and maintenance overhead
- Reinventing solved problems
- Opportunity cost of not using proven solutions
