# ADR-002: Documentation Versioning Strategy

## Status

**Accepted**

## Context

As the Phenotype ecosystem grows, maintaining documentation for multiple versions of libraries, APIs, and tools becomes critical. Users need to access documentation that matches the version they are using, and developers need to update documentation without breaking links for existing users.

Key challenges:

1. **Multi-Version Support**: Different projects release at different cadences, requiring independent versioning
2. **Cross-Version Links**: Documentation often references other components that may have different versions
3. **URL Stability**: Bookmarks and external links must remain valid across version changes
4. **Search Engine Optimization**: Search results should direct users to appropriate versions
5. **Maintenance Overhead**: Managing multiple documentation versions without excessive duplication
6. **Deprecation Strategy**: Clear communication when features and their documentation are deprecated

## Decision

We will implement a unified versioning strategy with the following components:

### 1. Semantic Versioning Alignment

**Rationale**: Documentation versions align with software versions for clarity.

**Decision Details**:
- Documentation versions follow the same MAJOR.MINOR.PATCH format as the software
- MAJOR version changes trigger new documentation versions
- MINOR versions may add features documented in the same major version stream
- PATCH versions update documentation inline without versioning
- Pre-release versions (alpha, beta, rc) are documented with clear warnings

### 2. URL Structure

**Rationale**: Predictable URLs enable bookmarking and external linking.

**Decision Details**:
- Format: `/docs/{project}/{version}/{path}`
- Latest stable version accessible at `/docs/{project}/latest/{path}`
- Unversioned URLs redirect to latest stable
- Version selector UI component on all versioned pages
- Canonical URLs point to latest version for SEO

### 3. Version Branching Strategy

**Rationale**: Git branching must support parallel documentation maintenance.

**Decision Details**:
- Main branch always contains latest development documentation
- Release branches (e.g., `release/v1.x`) maintain documentation for active versions
- Documentation changes for released versions cherry-picked to release branches
- Archived versions remain available but marked as unmaintained
- Version compatibility matrix maintained for cross-project references

### 4. Content Deduplication

**Rationale**: Minimize maintenance overhead for shared content.

**Decision Details**:
- Shared concepts documented once in "concepts" section
- Version-specific differences handled via content tabs or inline notes
- API reference generated from source code, ensuring version accuracy
- Changelogs aggregate changes across versions
- "Since version X.Y" annotations for feature documentation

### 5. Deprecation Workflow

**Rationale**: Clear communication prevents user confusion.

**Decision Details**:
- Deprecated features marked with warning callouts
- Migration guides maintained for two major versions
- Deprecated documentation redirects to replacement or migration guide
- End-of-life schedule published for documentation versions
- RSS/Atom feed for documentation updates

## Consequences

### Positive

1. **User Clarity**: Users always see documentation matching their installed version
2. **Link Stability**: URLs remain valid across version changes via redirects
3. **SEO Optimization**: Canonical URLs and version sitemaps improve search rankings
4. **Reduced Confusion**: Clear deprecation notices prevent using outdated features
5. **Parallel Development**: Multiple versions can be maintained simultaneously
6. **Audit Trail**: Version history provides documentation change tracking

### Negative

1. **Storage Costs**: Each version requires storage space
2. **Build Complexity**: CI/CD must handle multiple version builds
3. **Search Complexity**: Search index must handle versioned content
4. **Maintenance Overhead**: Bug fixes may need application across multiple versions
5. **Decision Fatigue**: Users must select appropriate version from dropdown

### Neutral

1. **URL Length**: Versioned URLs are longer than unversioned
2. **Cache Management**: Multiple versions require careful cache invalidation
3. **Analytics**: Versioned analytics provide usage insights per version

## Alternatives

### Alternative 1: Single Latest Version Only

**Considered**: Only maintain documentation for the latest version.

**Rejected Because**:
- Users on older versions have no reference documentation
- Breaking changes become extremely disruptive
- Enterprise users often lag behind latest versions
- Cannot provide migration guidance between versions

### Alternative 2: Date-Based Versioning

**Considered**: Version documentation by release date instead of semantic version.

**Rejected Because**:
- Harder to correlate with software versions
- Dates don't convey breaking change information
- Less intuitive for semantic versioning users

### Alternative 3: Feature Flags in Single Document

**Considered**: Single document with feature flags showing/hiding version-specific content.

**Rejected Because**:
- Documents become unwieldy with many versions
- Performance impact of rendering all versions
- SEO challenges with hidden content
- Difficult to maintain accurate version compatibility

### Alternative 4: Separate Documentation Repositories

**Considered**: Each version in its own repository.

**Rejected Because**:
- Cross-version linking becomes difficult
- Multiplied maintenance overhead
- No easy way to compare versions
- Fragmented contribution process
