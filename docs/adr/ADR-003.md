# ADR-003: Cross-Project Linking Standards

## Status

**Accepted**

## Context

The Phenotype ecosystem consists of numerous interconnected projects. Documentation frequently references concepts, APIs, and features from other projects. Without standardized linking, documentation becomes fragmented, links break during refactoring, and users cannot navigate effectively across project boundaries.

Key challenges:

1. **Link Rot**: URLs change as projects evolve, breaking cross-project references
2. **Circular Dependencies**: Documentation links can create circular reference chains
3. **Context Loss**: Linking to external documentation may confuse users about which project they're viewing
4. **Offline Access**: Documentation builds need to handle missing external resources gracefully
5. **Version Compatibility**: Links between projects must respect version compatibility matrices
6. **SEO Impact**: Cross-linking strategy affects search engine ranking and discoverability

## Decision

We will implement standardized cross-project linking with the following architecture:

### 1. Link Identifier System

**Rationale**: Stable identifiers decouple links from URL structure.

**Decision Details**:
- All linkable content receives a unique identifier: `{project}:{category}:{slug}`
- Categories: `concept`, `api`, `guide`, `tutorial`, `reference`
- Slugs derived from document title, manually overrideable for stability
- Identifiers stored in frontmatter metadata
- Link resolution service maps identifiers to current URLs

### 2. Link Resolution Architecture

**Rationale**: Dynamic resolution handles URL changes transparently.

**Decision Details**:
- Build-time link resolution for static generation
- Link manifest JSON files per project version
- Central registry service for runtime resolution (optional)
- Fallback chain: specific version -> latest stable -> error page with suggestions
- Validation CI job fails builds with unresolved cross-project links

### 3. Link Syntax Standards

**Rationale**: Consistent syntax enables tooling and validation.

**Decision Details**:
- Short links: `[text](xref:project:category:slug)`
- Versioned links: `[text](xref:project:v1.2:category:slug)`
- Auto-generated links: `[](autoxref:project:category:slug)` uses target title as text
- External links: Standard Markdown with validation warnings
- Anchor links: `[text](xref:project:category:slug#section-id)`

### 4. Link Quality Assurance

**Rationale**: Automated validation prevents broken links.

**Decision Details**:
- Pre-commit hooks check link syntax validity
- CI pipeline validates all cross-project links against registry
- Weekly scheduled link audit with reports
- Broken link alerts sent to owning project teams
- Link health dashboard for documentation maintainers

### 5. Navigation Context Preservation

**Rationale**: Users need awareness when crossing project boundaries.

**Decision Details**:
- Cross-project links open in same tab with project context banner
- Banner shows: target project name, version, "Return to {source}" link
- Breadcrumb navigation updated to show project transition
- Optional: Modal preview of external content before navigation
- Related projects sidebar widget on concept pages

## Consequences

### Positive

1. **Link Stability**: Identifier-based links survive URL restructuring
2. **Better UX**: Context preservation reduces user confusion
3. **Maintainability**: Automated validation catches issues early
4. **Discoverability**: Cross-linking improves content discovery
5. **SEO Benefits**: Proper canonical links improve search rankings
6. **Refactoring Safety**: Structural changes don't break external references

### Negative

1. **Complexity**: Link resolution adds build complexity
2. **Performance**: Link validation increases build time
3. **Learning Curve**: New syntax requires documentation contributor training
4. **Registry Dependency**: Link resolution depends on registry availability
5. **Migration Effort**: Existing links need conversion to new system

### Neutral

1. **Identifier Management**: Requires governance of identifier uniqueness
2. **Version Coordination**: Cross-version links need careful management
3. **Caching Strategy**: Link manifests benefit from aggressive caching

## Alternatives

### Alternative 1: Direct URLs Only

**Considered**: Use standard direct URLs for all cross-project links.

**Rejected Because**:
- URL changes break all references
- No validation possible during build
- Difficult to track link dependencies
- Users lose context when navigating

### Alternative 2: Wiki-Style Links

**Considered**: Use double-bracket syntax like `[[Project:Page]]`.

**Rejected Because**:
- Non-standard Markdown extension
- Conflicts with some Markdown flavors
- Harder to implement fallback behavior

### Alternative 3: Relative Path Links

**Considered**: Use relative paths assuming unified documentation structure.

**Rejected Because**:
- Requires all projects in monorepo
- Doesn't work with external project references
- Breaks when restructuring documentation hierarchy

### Alternative 4: Manual Link Management

**Considered**: Track all cross-project links manually in a registry.

**Rejected Because**:
- High maintenance overhead
- Easy to miss during updates
- No automation possible
- Scale limitations
