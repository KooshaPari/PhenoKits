# ADR-004: Content Reuse and Single-Sourcing Strategy

## Status

**Accepted**

## Context

The Phenotype ecosystem documentation contains significant duplication. Concepts, API patterns, and getting started instructions appear across multiple project documentations. This duplication creates maintenance burden and risk of inconsistency when updates occur.

Key challenges:

1. **Content Duplication**: Same concepts explained differently in different projects
2. **Update Propagation**: Fixing an error requires updates in multiple locations
3. **Translation Overhead**: Multiple copies multiply translation costs
4. **Consistency**: Different writing styles create fragmented experience
5. **Scope Creep**: Projects document things outside their scope because dependencies lack documentation
6. **Include Complexity**: Technical solutions for content reuse can be complex to maintain

## Decision

We will implement a single-sourcing strategy with the following components:

### 1. Shared Content Repository

**Rationale**: Centralized shared content reduces duplication.

**Decision Details**:
- `docs/shared/` directory contains cross-cutting documentation
- Organized by topic: concepts, patterns, getting-started, glossary
- Versioned alongside main documentation
- All projects can reference shared content
- Contribution process for adding shared content

### 2. Content Inclusion System

**Rationale**: Technical inclusion enables single-sourcing without copy-paste.

**Decision Details**:
- VitePress custom container for transclusion: `:::include{path="shared/concept.md"}`
- Variable substitution for project-specific customization
- Conditional content based on project metadata
- Partial includes for specific sections
- Build-time resolution with caching

### 3. Content Taxonomy

**Rationale**: Classification enables discovery and appropriate reuse.

**Decision Details**:
- Every document tagged with: scope, audience, topic, maturity
- Scope: project-specific, shared-concept, ecosystem-wide
- Audience: beginner, intermediate, advanced, contributor
- Topic: installation, configuration, development, deployment
- Maturity: draft, review, stable, deprecated

### 4. Contribution Guidelines

**Rationale**: Clear rules prevent inappropriate content reuse.

**Decision Details**:
- Shared content must be truly cross-cutting (used by 3+ projects)
- Project-specific examples belong in project docs
- Shared content maintained by documentation working group
- Regular audits identify content that should be shared or split
- Style guide enforced via automated linting

### 5. Generation Pipeline

**Rationale**: Automation ensures consistency and reduces manual work.

**Decision Details**:
- API documentation generated from source code
- Changelogs aggregated from commit messages
- Configuration examples validated against schemas
- Diagrams generated from text descriptions (Mermaid, PlantUML)
- Screenshots automated via Playwright

## Consequences

### Positive

1. **Consistency**: Shared content ensures uniform explanations
2. **Maintainability**: Updates in one place apply everywhere
3. **Reduced Duplication**: Less content to maintain and translate
4. **Clear Ownership**: Shared content has explicit maintainers
5. **Quality**: Shared content receives more review attention
6. **Scale**: New projects leverage existing documentation

### Negative

1. **Complexity**: Inclusion system adds build complexity
2. **Coupling**: Shared content changes affect multiple projects
3. **Performance**: Resolving includes increases build time
4. **Debugging**: Tracing content sources can be difficult
5. **Governance**: Requires coordination across project teams

### Neutral

1. **Versioning**: Shared content versioning must align with dependent projects
2. **Caching**: Include resolution benefits from aggressive caching
3. **Tooling**: Custom VitePress plugins required for advanced features

## Alternatives

### Alternative 1: Manual Copy-Paste

**Considered**: Copy content between projects and update manually.

**Rejected Because**:
- High maintenance overhead
- Guaranteed inconsistency over time
- No way to track which projects have which content
- Translation nightmare

### Alternative 2: Git Submodules

**Considered**: Use Git submodules for shared documentation.

**Rejected Because**:
- Submodule complexity well-documented as problematic
- Version synchronization challenges
- Steep learning curve for contributors
- CI/CD complications

### Alternative 3: External Documentation Service

**Considered**: Use a headless CMS for content management.

**Rejected Because**:
- Adds external dependency
- More complex authoring workflow
- Vendor lock-in concerns
- Developer experience degraded vs. Git-based workflow

### Alternative 4: Symbolic Links

**Considered**: Use file system links for shared content.

**Rejected Because**:
- Poor Windows support
- Git handling of symlinks is complex
- No build-time customization possible
- Editor support varies
