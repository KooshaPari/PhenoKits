# ADR-005: Documentation Search Architecture

## Status

**Accepted**

## Context

Effective search is critical for documentation usability. Users must quickly find relevant information across a growing corpus of documentation spanning multiple projects, versions, and content types. The current state-of-the-art documentation search combines multiple techniques to provide fast, relevant results.

Key challenges:

1. **Scale**: Documentation corpus growing rapidly across projects
2. **Relevance**: Generic search often returns irrelevant technical results
3. **Speed**: Search must feel instantaneous for good UX
4. **Version Handling**: Search should respect version context
5. **Offline Support**: Some environments require offline documentation
6. **Multi-Language**: Search across translated content
7. **Accessibility**: Search must work with assistive technologies

## Decision

We will implement a hybrid search architecture with the following components:

### 1. Tiered Search Strategy

**Rationale**: Different scopes require different search implementations.

**Decision Details**:
- **Local Search**: VitePress built-in FlexSearch for single-project docs
- **Federated Search**: Algolia DocSearch for multi-project portal
- **IDE Integration**: LSP-based symbol search for API reference
- **Command Palette**: Quick navigation within documentation site
- **Global Search**: Custom Elasticsearch for advanced queries

### 2. Index Architecture

**Rationale**: Proper indexing enables fast, relevant retrieval.

**Decision Details**:
- Content indexed at build time for static sites
- Incremental indexing for large documentation sets
- Metadata enrichment: headings, code blocks, tags
- Semantic search vectors for concept matching
- Version-specific indexes with fallback handling

### 3. Relevance Scoring

**Rationale**: Technical documentation requires domain-specific ranking.

**Decision Details**:
- Title matches weighted higher than body
- Exact phrase matches rank above word matches
- Recently updated content boosted slightly
- User context (current project/version) influences ranking
- API reference pages prioritized for symbol queries
- Heading hierarchy contributes to context matching

### 4. Search UI Components

**Rationale**: Search interface significantly impacts usability.

**Decision Details**:
- Modal search with keyboard shortcut (Cmd/Ctrl+K)
- Instant results as user types
- Grouped results by content type
- Preview snippets with highlighting
- Breadcrumb context for each result
- Recent searches and suggestions
- Filters: project, version, content type

### 5. Analytics and Improvement

**Rationale**: Data-driven improvements to search quality.

**Decision Details**:
- Search query logging (privacy-respecting)
- Click-through rate tracking per result
- Failed search detection and reporting
- Popular queries identification
- Content gap analysis from failed searches
- Regular relevance tuning based on metrics

## Consequences

### Positive

1. **User Efficiency**: Fast, relevant search reduces time-to-information
2. **Discovery**: Cross-project search reveals related content
3. **Self-Service**: Good search reduces support burden
4. **Insights**: Analytics reveal documentation gaps
5. **Accessibility**: Keyboard-first design benefits all users
6. **Mobile Support**: Search works across device types

### Negative

1. **Build Complexity**: Index generation adds build time
2. **Infrastructure**: Federated search requires service dependencies
3. **Privacy**: Search analytics must be carefully implemented
4. **Maintenance**: Search relevance requires ongoing tuning
5. **Cost**: Commercial search services incur expenses

### Neutral

1. **Index Size**: Large documentation requires efficient index compression
2. **Cache Strategy**: Search indexes benefit from CDN caching
3. **Real-time**: Static search indexes don't include real-time changes

## Alternatives

### Alternative 1: Client-Side Only

**Considered**: Use only VitePress built-in FlexSearch for all documentation.

**Rejected Because**:
- Doesn't scale to very large documentation sets
- No cross-project search capability
- Limited relevance tuning options
- Large download size for comprehensive indexes

### Alternative 2: External Search Engine Only

**Considered**: Rely entirely on Google Custom Search or similar.

**Rejected Because**:
- No control over ranking algorithm
- Privacy concerns with external services
- Limited customization options
- Potential cost at scale

### Alternative 3: Database-Backed Search

**Considered**: Store documentation in database with full-text search.

**Rejected Because**:
- Conflicts with docs-as-code approach
- Adds operational complexity
- Slower than static index solutions
- More expensive to operate

### Alternative 4: AI-Powered Chat

**Considered**: Replace search with conversational AI interface.

**Rejected Because**:
- Higher latency than traditional search
- Hallucination risks for technical accuracy
- Higher operational costs
- Complements rather than replaces search
