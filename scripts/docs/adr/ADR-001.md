# ADR-001: Scripting Language Strategy

**Status**: Accepted

**Date**: 2026-04-04

**Context**: The scripts/ directory contains both shell scripts and Python scripts. We need to establish clear guidelines for when to use each language, considering portability, dependencies, maintainability, and team expertise.

---

## Decision Drivers

| Driver | Priority | Notes |
|--------|----------|-------|
| Portability | High | Must work across macOS, Linux, and CI environments |
| Dependencies | High | Minimize external tool installation requirements |
| Maintainability | High | Team primarily Python-experienced |
| Performance | Medium | Scripts are typically short-running |
| Tool availability | High | Python available everywhere, bash varies |

---

## Options Considered

### Option 1: Shell-First Approach

**Description**: Prefer bash for all scripts, using Python only when shell limitations require it.

**Pros**:
- Native OS integration
- No interpreter installation needed
- Standard UNIX tools available
- Fast startup for simple operations

**Cons**:
- Complex string manipulation is error-prone
- Inconsistent behavior across platforms
- Limited library ecosystem
- Harder to test reliably

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Startup time | <10ms | Measured |
| Simple ops | 0.1-0.5s | Typical |
| String processing | Slow | Benchmarks |

### Option 2: Python-First Approach

**Description**: Prefer Python for all scripts, using shell for simple wrappers only.

**Pros**:
- Consistent behavior across platforms
- Rich standard library
- Easy testing with pytest
- Better error handling
- Team expertise alignment

**Cons**:
- Requires Python installation
- Slightly higher startup overhead
- Some OS operations require subprocess

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Startup time | 50-150ms | Measured |
| Complex logic | Fast | O(n) typically |
| Stdlib coverage | Excellent | Python stdlib |

### Option 3: Hybrid Approach (Selected)

**Description**: Use shell for simple wrappers and OS-level operations; use Python for complex logic, parsing, and anything requiring testing.

**Pros**:
- Best of both worlds
- Shell for what it's best at (process spawning, pipes)
- Python for complex logic
- Gradual migration path

**Cons**:
- Multiple language maintenance
- Team must know both
- Boundary decisions can be subjective

**Performance Data**:
| Metric | Shell | Python | Notes |
|--------|-------|--------|-------|
| File I/O | Fast | Fast | Similar |
| HTTP requests | curl | requests/httpx | Python more flexible |
| JSON parsing | jq | json module | Python simpler |
| Process control | Excellent | subprocess | Shell native |

---

## Decision

**Chosen Option**: Option 3 - Hybrid Approach

**Rationale**: The hybrid approach optimizes for both capability and simplicity. Shell excels at process orchestration and piping; Python excels at complex logic, data manipulation, and testing. This matches the current script inventory where simple wrappers are shell and complex tools (traceability-check.py) are Python.

**Evidence**:
- Existing scripts follow this pattern naturally
- Industry standard (Google, Facebook, Netflix) use similar approach
- Team has strong Python expertise
- CI environments always have both available

---

## Implementation Guidelines

### Use Shell When:
- Script is <20 lines and primarily orchestrates other tools
- Heavy use of pipes and process substitution
- Simple file existence checks
- Wrapper around existing CLI tools

### Use Python When:
- Script logic exceeds simple linear flow
- Requires data structure manipulation (dicts, lists)
- Needs to parse structured formats (JSON, YAML)
- Requires testing infrastructure
- Complex string manipulation beyond regex

### Boundary Examples:

| Task | Language | Reason |
|------|----------|--------|
| Check file exists, exit if missing | Shell | Simple, linear |
| Parse JSON, validate schema | Python | Complex parsing |
| Format and send report | Python | Data transformation |
| Spawn process, pipe output | Shell | Native capability |
| Multi-repo traceability check | Python | Complex logic, testing needed |

---

## Consequences

### Positive

- Optimal tool for each job
- Easier testing for complex scripts
- Better error messages in Python
- Maintainable long-term

### Negative

- Two languages to maintain
- Style guidelines needed for both
- Potential boundary disputes

### Neutral

- Gradual migration ongoing
- Some legacy shell scripts remain
