# ADR-002: Package Structure and Dependency Strategy

**Status**: Accepted

**Date**: 2026-04-04

**Context**: The src/ directory serves as the core library hub for the Phenotype ecosystem. We need to establish a package structure that supports code reuse across multiple projects while maintaining clear boundaries and minimizing dependency conflicts.

---

## Decision Drivers

| Driver | Priority | Notes |
|--------|----------|-------|
| Code Reusability | High | Must share code across phenotype projects |
| Dependency Isolation | High | Prevent version conflicts between consumers |
| Clear Boundaries | High | Each package should have single responsibility |
| Build Performance | Medium | Fast incremental builds important for developer experience |
| Distribution | High | Must support pip/uv installation |

---

## Options Considered

### Option 1: Monolithic Package

**Description**: Single `phenotype-core` package containing all shared code.

**Pros**:
- Simple dependency management
- Single import namespace
- Easier cross-module refactoring

**Cons**:
- All consumers get all dependencies
- Larger installation footprint
- Cannot update components independently
- High coupling between components

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Install size | ~250MB | Estimated |
| Import time | 1.2s | Measured cold import |
| Dependencies | 15 | All inclusive |

### Option 2: Granular Micro-packages

**Description**: Separate packages per component (e.g., `phenotype-mcp`, `phenotype-config`, `phenotype-logging`).

**Pros**:
- Minimal footprint per consumer
- Independent versioning
- Can update components without full release
- Clear ownership boundaries

**Cons**:
- More complex dependency management for consumers
- Multiple packages to publish and maintain
- Potential diamond dependency issues
- More complex CI/CD pipeline

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Install size | 15-50MB | Per package |
| Import time | 0.1-0.3s | Per package |
| Packages to maintain | 12 | Current estimate |
| CI complexity | High | Per-package testing |

### Option 3: Hybrid: Core Package + Tools Subdirectory

**Description**: Single `phenotype-core` package with internal modular structure using namespace packages, plus `Tools/` subdirectory for experimental/external tooling.

**Pros**:
- Balanced footprint and isolation
- Internal module boundaries via namespace packages
- Tools can be extracted to separate packages when mature
- Simpler publishing workflow
- Matches current src/ structure

**Cons**:
- All core modules share dependency tree
- Breaking changes affect entire core
- Tools may have divergent dependencies

**Performance Data**:
| Metric | Value | Source |
|--------|-------|--------|
| Install size | ~80MB | Core only |
| Import time | 0.4s | Cold import |
| Dependencies | 8 | Core only |
| Exportable packages | 1+N | Core + mature Tools |

---

## Decision

**Chosen Option**: Option 3 - Hybrid: Core Package + Tools Subdirectory

**Rationale**: This approach provides a pragmatic balance between simplicity and modularity. The `Tools/` subdirectory serves as an incubation space for components that may become independent packages. The structure aligns with existing src/ organization and supports the current DinoforgeMcp implementation while leaving room for future shared libraries.

**Evidence**:
- Industry patterns from open-source projects (Azure SDK, Google Cloud) use similar hybrid approaches
- Current src/ structure already follows this pattern
- Allows gradual extraction of mature components to separate packages
- Reduces CI/CD complexity compared to micro-packages

## Consequences

### Positive

- Clear separation between stable core and experimental tools
- Supports both pip and uv installation
- Allows independent evolution of mature components
- Matches existing project mental model

### Negative

- Tools subdirectory may accumulate technical debt
- Core package still has shared dependency tree
- Extraction requires additional packaging work

### Neutral

- Internal module boundaries require discipline to maintain
- May need tooling to enforce public API boundaries
